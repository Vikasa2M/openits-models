# NATS reference profile

This directory holds the **NATS reference binding** for the OpenITS
model layer: the concrete transport, envelope, and subject grammar the
in-tree reference implementation ships. It is one binding among several
possible ones ‚Äî the YANG models themselves are transport-neutral (see
[`docs/data-model.md`](../../docs/data-model.md), "Encodings and
transports: the schema is the contract"). Nothing in `yang/`,
`api/proto/`, or `pkg/` depends on anything described here.

Conformance to this profile is **Tier 2** conformance in
[`docs/07-conformance.md`](../../docs/07-conformance.md). Tier 1 (model
conformance) is transport-neutral and can be claimed by a deployment
that never touches NATS. This profile is what "OpenITS-compatible" adds
on top today, and it is the template a future MQTT/Kafka/RESTCONF
profile would follow as a sibling.

## Subject grammar

Every event is published on a **seven-token** NATS subject:

```
openits.{region}.{agency}.{agency-unit}.{service}.{controller-id}.{event}
```

| Token | Meaning |
|-------|---------|
| `openits` | Fixed authority prefix. |
| `{region}` | Region identifier, e.g. `us-tx`. |
| `{agency}` | Owning agency, e.g. `txdot`. |
| `{agency-unit}` | Sub-unit / district, e.g. `d07`. |
| `{service}` | Service name, e.g. `signal-control`, `dms`, `rsu`. |
| `{controller-id}` | Device / controller identifier, e.g. `i35-exit-214`. |
| `{event}` | Event leaf, the notification's last name token, e.g. `fault-raised`. |

Each token is lowercase alphanumeric with hyphens (`^[a-z0-9][a-z0-9-]*$`).
The `{region}`, `{agency}`, and `{agency-unit}` tokens are validated
against the **agency registry** ‚Äî they are not free-form. A consumer
subscribes with NATS wildcards over this hierarchy, e.g. `openits.>`
for everything, or
`openits.us-tx.txdot.d07.signal-control.>` for one district's signals.

Example:

```
openits.us-tx.txdot.d07.signal-control.i35-exit-214.fault-raised
```

## CloudEvents envelope

Messages use **CloudEvents 1.0 binary mode**: the CloudEvents
attributes ride as `ce-*` message headers and the body is the raw
encoded payload. The required header set:

| Header | Value |
|--------|-------|
| `ce-specversion` | `1.0`. |
| `ce-type` | `openits.<service>.<event>.v<major>`, e.g. `openits.signal-control.fault-raised.v1`. The `v<major>` tracks the ce-type major version. |
| `ce-source` | `urn:openits:<entity-kind>:<region>:<agency>:<unit>:<id>` ‚Äî a well-formed OpenITS source URN identifying the emitting device or service instance. |
| `ce-id` | Event identifier, unique within the source. The reference binding derives it deterministically from event content so a retried publish carries the **same** `ce-id`, letting consumers deduplicate without coordination. |
| `ce-time` | RFC 3339 / ISO 8601 timestamp. |
| `ce-dataschema` | URL of the immutable schema-registry snapshot the body validates against, e.g. `https://schemas.openits.vikasa.io/openits-common-fault-events/<revision>/`. |
| `ce-datacontenttype` | Media type of the body ‚Äî `application/protobuf` for the reference binding's per-event Protobuf encoding. |

`traceparent` (W3C Trace Context) is optional but recommended.

The deterministic-`ce-id` idempotency rule is worth calling out because
its *root* is model-level, not transport-level: event identity and
dedup semantics come from the YANG event-header / sequence semantics.
The reference binding carries that identity in the `ce-id` header;
another binding would carry the same identity in whatever field its
envelope provides. Model conformance (Tier 1) is where the
"retried publish produces the same event id" rule is asserted; this
profile only fixes *where* the id is written on the wire.

## Encoding

The reference encoding is **per-event Protobuf**, generated from the
YANG via ygot (`ce-datacontenttype: application/protobuf`). The body is
the Protobuf-serialized notification message. A binding is free to
serve the same notification as RFC 7951 JSON, CBOR, or XML validated
against the same schema-registry snapshot; the model does not mandate
Protobuf. See the "same data, three ways" worked example in
[`docs/data-model.md`](../../docs/data-model.md).

## JetStream

The reference transport is **NATS JetStream**. Streams capture the
`openits.>` subject space so consumers get durable, replayable
delivery rather than fire-and-forget core NATS. Delivery and cadence
carriage ‚Äî at-least-once delivery, per-consumer acknowledgement, and
the redelivery behavior that makes the deterministic `ce-id` load-
bearing ‚Äî are properties of this profile, not of the model layer. A
consumer that replays a stream relies on `ce-id` stability to
deduplicate the redelivered events.

## AsyncAPI

`asyncapi.yaml` in this directory is the machine-readable description of
this profile's pub/sub surface: one channel/operation/message per
ce-type, each message payload embedded as that notification's JSON
Schema (not a URL pointer). It is generated in-repo from the same
YANG-derived ce-type catalog as the reference binding ‚Äî `make asyncapi`
‚Äî so it never drifts from the schema. It is a NATS-profile artifact:
the channels correspond to the subject grammar above, and the messages
carry the CloudEvents headers described here. See
[Design decisions](../../docs/04-design-decisions.md#generated-asyncapi-not-hand-maintained)
for why it is generated rather than hand-maintained.

## Tier 2 conformance tooling

The conformance harness (`tools/conformance/`) mixes Tier 1 and Tier 2
checks in one run; the ones that validate **this profile** are:

- **Subject grammar** ‚Äî `tools/conformance/tests/subject.go`:
  `TestSubject_SevenTokenShape` and `TestSubject_OpenitsPrefix` assert
  the seven-token shape, the `openits.` authority prefix, and
  lowercase-alnum-hyphen tokens.
- **CloudEvents envelope** ‚Äî `tools/conformance/tests/cetype.go`:
  `TestCEType_OpenitsFormat` (ce-type shape),
  `TestCESource_URN` (well-formed `urn:openits:` source), and
  `TestCEID_Present` (a `ce-id` is present so replay can deduplicate).

The **per-kind shape checks** in the same harness (cadence, lifecycle,
per-event payload assertions in `tools/conformance/tests/<service>.go`)
are Tier 1 ‚Äî they assert model-level behavior and hold under any
binding. They select events by the subject's last token only, which is
the model-level event name; they do not depend on the transport. Those
checks stay with the model layer.

The `scripts/check-naming.sh` guard also rejects the legacy five-token
subject hierarchy and the pre-OpenITS schema-URN and ce-type forms in
favor of the shapes above; it is a repository-hygiene gate for this
profile's conventions.

## When this profile moves out

This profile stays in `openits-models` until one of the following is
true, at which point it moves to its own repository consuming released
model bundles:

- a **second binding profile** exists (MQTT, Kafka, RESTCONF, ‚Ä¶), so the
  model repo would otherwise privilege one transport;
- an **external implementer needs profile conformance without the model
  toolchain** (i.e. wants the profile spec + AsyncAPI without cloning
  the YANG/Go generation stack);
- the **profile needs its own release cadence**, decoupled from model
  revisions.

Until then it lives here so the reference binding and the model it
binds evolve together in one change set.
