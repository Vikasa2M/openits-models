# AsyncAPI ce-type delta: collector's 72 → derived catalog's 91

`asyncapi.yaml` is now generated in-repo (`make asyncapi`, P2b-2 Task 2) from
the YANG-derived ce-type catalog (`tools/yang-proto-gen`: `BuildCatalog`),
replacing the previous file — generated by a now-obsolete tool in the
collector repo (`tools/asyncapi-gen`, sourcing `internal/cloudevents`) and
committed here as a static, pre-v1 snapshot. This is a **set difference**
between that old snapshot's 72 ce-types and the newly-derived catalog's 91
ce-types, captured as part of the regeneration (Task 2, Step 4 of the
[P2b-2 plan](superpowers/plans/2026-07-11-p2b2-asyncapi.md)).

Both sets were extracted with the same method, for an apples-to-apples
comparison:

```bash
git show HEAD:asyncapi.yaml | grep -oE 'openits\.[a-z-]+\.[a-z-]+\.v[0-9]+' | sort -u   # old, captured pre-regeneration
grep -oE 'openits\.[a-z-]+\.[a-z-]+\.v[0-9]+' asyncapi.yaml | sort -u                    # new, post make asyncapi
```

Old: 72 distinct ce-types. New: 91. Net +19 (23 added, 4 removed).

## 23 ADDED — common events now fanned to more services

All 23 additions are notifications that already existed in the YANG (and so
were always emitted for *some* services), now fanned to services the
collector's hand-maintained catalog hadn't caught up to. Two categories:

### Common-events fan-out (18 ce-types)

`openits-common-fault-events` and `openits-common-comm-health-events`
notifications fan to every service whose `<service>-event-kind` identity
root has a descendant under the notification's `kind` behavioral base
(`fault-event-kind` / `comm-health-event-kind`) — see `BuildCatalog`'s
derivation rule in `tools/yang-proto-gen/catalog.go`. The old collector
catalog was frozen before P3c/P3c-2 connected `gateway`, `perception`,
`traffic-sensor`, and `reversible-lane` into the fault taxonomy (and before
`comm-health-event-kind` had any service sub-bases at all), so it never
fanned these notifications to those services:

| ce-type | old fan-out | new fan-out | added here |
|---|---|---|---|
| `fault-raised` | dms, ess, ramp-metering, rsu, signal-control (5) | + gateway, perception, reversible-lane, traffic-sensor (9) | 4 |
| `fault-cleared` | dms, ess, ramp-metering, rsu, signal-control (5) | + gateway, perception, reversible-lane, traffic-sensor (9) | 4 |
| `controller-fault-event` | signal-control only (1) | all 9 services | 8 |
| `comm-health-event` | none (0) | gateway, signal-control (2) | 2 |

### New per-service module coverage (5 ce-types)

`reversible-lane` has no entries at all in the old catalog (0 of 72) — the
collector never onboarded this service's own
`openits-reversible-lane-events` module. Its 5 service-specific

`openits.reversible-lane.lcs-conflict-detected.v1`,

(The other 3 of reversible-lane's 8 new ce-types —

## 4 REMOVED — stale collector entries with no YANG notification

Four ce-types in the old catalog have **no corresponding YANG
`notification`** anywhere in `yang/` today (`grep -rl` for each event name
across every module returns nothing):

- `openits.signal-control.operational-status.v1`
- `openits.signal-control.detector-report.v1`
- `openits.signal-control.plan-applied.v1`
- `openits.rsu.rsu-broadcast-sample.v1`

These read as gen-1 collector artifacts: periodic state-snapshot "events"
the collector synthesizes from polled device state (e.g.
`operational-status` looks like a serialization of a `config false`
container, not a modeled `notification`) rather than notifications the YANG
schema actually declares. Because `BuildCatalog` derives its catalog
strictly from declared YANG notifications, it cannot — and should not —
reproduce a ce-type that has no schema behind it.

One line: **these deltas (the 23 additions and the 4 unmodeled removals)
track the remaining gen-1→gen-2 collector migration** — reconciling
the collector's event surface to this derived catalog (fanning the new
common events, onboarding reversible-lane, and either modeling
`operational-status`/`detector-report`/`plan-applied`/
`rsu-broadcast-sample` as real notifications or retiring them) is tracked
there, not in this repo.

## Slice 3: 91 → 77 — bespoke per-service fault/mode notifications deleted Slice 3 (dated after the 72→91 regeneration recorded above) deletes
the bespoke per-service `fault-raised` / `fault-cleared` / `mode-changed`
notifications that Slice 1/2 identified as the anti-pattern this
consolidation removes. Every one of the 91 ce-types from the section above
is either unaffected or replaced by the equivalent common ce-type
(`openits.<service>.fault-raised.v1`, `.fault-cleared.v1`,
`.mode-changed.v1`) that already fans out from
`openits-common-fault-events` / `openits-common-mode-events` via the
`<service>-fault-event-kind` / `<service>-mode-event-kind` taxonomy
(`derived-from-or-self(kind, 'openits-types:fault-event-kind')` etc. —
see [the data model](data-model.md#the-event-taxonomy)). Net effect:
**91 → 77, exactly 14 removed, zero added.**

```bash
git show ce4095e:asyncapi.yaml | grep -oE 'openits\.[a-z-]+\.[a-z-]+\.v[0-9]+' | sort -u   # pre-Slice-3, 91 ce-types
grep -oE 'openits\.[a-z-]+\.[a-z-]+\.v[0-9]+' asyncapi.yaml | sort -u                       # post-Slice-3, 77 ce-types
```

### 4 ADDED — the collector state-snapshot events, now modeled

| ce-type | Task | Module |
|---|---|---|
| `openits.signal-control.operational-status-report.v1` | Task 1 | `openits-signal-control-events` |
| `openits.signal-control.detector-report.v1` | Task 2 | `openits-signal-control-events` |
| `openits.signal-control.plan-applied.v1` | Task 3 | `openits-signal-control-events` |
| `openits.rsu.rsu-broadcast-sample.v1` | Task 4 | `openits-rsu-events` |

These four back the collector's state-snapshot events: periodic or
on-change dumps of device/controller state that the gen-1 collector
synthesized from polling rather than from a modeled YANG notification.
Modeling them closes that gap — each is now a real `notification` with a
schema, so it gets a derived ce-type, a `schema.json`/`schema.proto` pair,
and a `field-numbers.yaml` entry like any other event.

Three keep the collector's original event name (`detector-report`,
`plan-applied`, `rsu-broadcast-sample`). The fourth does not:
`operational-status` is modeled as **`operational-status-report`** (design
decision D1). The notification carries a point-in-time snapshot of the
`config false` `state` container — a deliberate snapshot
event, not a synthesized poll result — so its ce-type intentionally
differs from the collector's old `operational-status` string. There is no
`openits.signal-control.operational-status.v1` ce-type in the catalog;
the collector migrates to `operational-status-report` to pick this event
up.

## final state (Slice 5): registry re-snapshotted to match 81

The full arc, end to end:

**72** (original collector catalog, `tools/asyncapi-gen`) → **91**
(taxonomy-derived catalog, first `BuildCatalog` regeneration — see "23
ADDED" / "4 REMOVED" above) → **77** (Slice 3 fault/mode consolidation —
see "Slice 3: 91 → 77" above) → **81** (Slice 4, state-snapshot
notifications modeled — see "Slice 4: 77 → 81" above).

81 is the current and final ce-type count. Slice 5 (this slice) makes
no catalog change — `make gen` and `make asyncapi` remain idempotent,
and `asyncapi.yaml`'s distinct `openits.<service>.<event>.v1` count is
still 81. What Slice 5 did was bring the **schema registry** into line
with this already-settled catalog: collapse the same-date stacked
revision blocks that accumulated across Slices 1-4 into one block per
module, delete the orphaned `-notifications`/`openits-signal-control-events`
snapshot dirs those slices left behind, and re-snapshot every affected
module (including the module added this slice for the new
operational-status-report notification) at its current revision. `make check-revisions` is green:
every module's latest revision is snapshotted and matches its
`schema.yang`.

Two catalog-adjacent items surfaced during are deliberately
**not** addressed here — they're deferred to standalone backlog issues
rather than folded into this consolidation:

- **** — `controller-fault-event` fans to all 9 services (the
  "23 ADDED" table above), a wider fan-out than the other common-fault
  notifications get; whether that's the taxonomy's intended shape or
  needs narrowing is its own question, tracked separately.
- **** — a `make validate-yang` coverage gap identified while
  planning this slice, unrelated to the ce-type catalog or the
  registry snapshots reconciled here. (taxonomy consolidation, Slices 1-5) is closed as of this
commit.

## (post-): controller-fault-event un-fanned, 81 → 73

The note above flagged `controller-fault-event` as fanning to all
9 services. resolves it. `controller-fault-event` is a
signal-controller HR-log fault (Indiana codes 173-185), not a
cross-service event, but it lived in `openits-common-fault-events` with
its `kind` based on the cross-service root `openits-types:fault-event-kind`
— so the fan-out rule matched every service. It is now **moved** to a
dedicated per-domain fault-events module with `kind`
narrowed to `sc-fault-event-kind`, so it routes to signal-control only
(catalog rule 1, module-prefix). `openits-common-fault-events` is left
holding only the genuinely cross-service `fault-raised`/`fault-cleared`.
(That dedicated module was itself later folded, along with the other
per-domain signal-control event modules, into the single
`openits-signal-control-events` by cut 3b.)

**8 REMOVED** (net 81 → 73), the spurious per-service fan-out:

```
openits.dms.controller-fault-event.v1
openits.ess.controller-fault-event.v1
openits.gateway.controller-fault-event.v1
```

`openits.signal-control.controller-fault-event.v1` is retained (now the

(`openits.common.v1` → `openits.signal_control.v1`) and its ce-type set
change. Registry re-snapshotted; `make check-revisions` green.
